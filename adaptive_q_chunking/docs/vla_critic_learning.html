VLA Critic Learning AQC-TD on frozen-VLA data

Adaptive Q-Chunking (ACSAC) critic trained on the insert-mouse-battery_annotated LeRobot dataset, using precomputed VLA latents and candidate action chunks. Companion to implementation_plan.html and aqc_distributional_guide.html.

1. Goal & setting

We verified the ACSAC critic on OGBench; now we move to the real target: an offline critic over a frozen VLA (Pi0.5). The policy is fixed; we only learn a value/critic that scores action chunks, for adaptive chunk-size selection and value-guided ranking. There is no environment rollout and no flow-BC actor — every action chunk we ever score is precomputed in the data.

2. The data

LeRobot v3.0, robot yam (bimanual, 14-DoF), 1119 episodes / 3.08M frames @ 60 fps, one task. Local copy under config.json's local path. Per-frame columns we use:

columnshaperole
rl_token(2048,)frozen-VLA latent → critic state token
action(14,)executed (behaviour) action → the chunk the trained Q conditions on
base_action(32, 50, 14)32 candidate chunks (H=50) sampled from the fine-tuned policy → the N candidates for the bootstrap max
reward()cost: −1e-4 living penalty/step + −0.5 terminal penalty on failure; range [−0.5, 0]
mc_return()discounted return-to-go, γ = 0.995 (verified mc_t = r_t + 0.995·mc_{t+1}); range [−0.5, 0]
episode_index, observation.commander_state—episode boundaries; teleop (expert) vs inference (policy rollouts incl. failures)
Value support is [−0.5, 0] (the exact mc_return range from meta/stats.json), far tighter than OGBench. With 51 atoms the bin width is 0.01.

3. The critic update (paper-faithful ACSAC)

For a transition starting at t, the per-prefix multi-step TD target (Eq. 6) for prefix length h is

$$ G_h \;=\; \sum_{j=0}^{h-1}\gamma^{j} r_{t+j} \;+\; \gamma^{h}\,\big[\, \text{valid}_h \cdot V(s_{t+h})\,\big] $$

where the bootstrap value is the EMaQ-style joint max over the N candidate chunks and the H prefix lengths (Eq. 19), aggregated by the ensemble min:

$$ V(s') \;=\; \min_{k\in[K]} \; \max_{n\in[N],\, h'\in[H]} \; Q_{\bar\phi}\!\big(s',\, a^{(n)}_{1:h'}\big), \qquad a^{(n)} = \texttt{base\_action}(s')^{(n)} $$

The critic predicts all H prefix values in one pass and is trained with HL-Gauss cross-entropy against the soft histogram of G_h over the fixed support.

Two data-specific touches

  • Candidates are precomputed. ACSAC samples N chunks from the policy at \(s'\); here base_action(s') already holds 32 such samples — so the joint max is computed directly on them, with no online sampling / flow rollout.
  • Terminal next-states. At the episode's last frame (the −0.5 failure penalty) there is no future, so we set \(V(s') = \texttt{mc\_return}(s')\) (= the terminal reward) instead of the candidate max. This is what teaches the critic that states near a failure are low-value.
Why the max (and not one chunk). Using a single chunk for the target would be SARSA — pure evaluation of the behaviour policy, identical to regressing to mc_return, with no improvement. The max over the 32 candidates is the only place the learned value can exceed BC; it is a γ-contraction whose fixed point is \(Q^{\pi^\star}\) (Appendix G), and it stays in-distribution (we only ever score policy-proposed chunks, never OOD actions).

4. Cost & the prefix-subsample knob

H=50, N=32 makes the bootstrap the dominant cost: faithfully bootstrapping at all H next-states is 32×50 = 1600 candidate-prefix critic evals per transition. We therefore bootstrap on a coarse prefix grid prefixes = [1, 10, 25, 50] (configurable) instead of all 50 — the critic still predicts all H prefixes, but the TD loss/bootstrap is only computed at the grid. This keeps the γ-contraction intent while cutting cost ≈ H/|grid|×. N can also be capped if memory is tight (the paper uses all 32).

5. Model capacity

The OGBench critic was tiny (n_embd = 128, 2 layers, ~2.5M params) for ~tens-of-dims states. The VLA state is a 2048-d latent, so we scale up the defaults:

knobOGBenchVLA default
head_dim → n_embd16 → 12848 → 384
num_layers24
mlp_dim5121024
num_atoms / support101 / data51 / fixed [−0.5, 0]
state_encoder_dims() (linear)() (linear) — ACSAC-faithful default

Bigger ≠ automatically better for value learning (overestimation/instability scale too), so width/depth should be a small sweep, not a fixed pick. Capacity is fully config-driven.

State token: linear by default (ACSAC-faithful). The 2048-d rl_token becomes the state token via a single linear Dense(2048 → n_embd) — exactly how ACSAC embeds the state in state-based tasks. (ACSAC's states are ~tens of dims, an expansion; here it's a 5× compression, so an MLP encoder may help — that's the opt-in vla_aqc_td_a51_stateenc preset with state_encoder_dims=(512,), to be confirmed against the linear baseline rather than assumed.)

7. Environment

Runs in the shared openpi uv (openpi-baseline_RLLAB/.venv, jax 0.5.3, flax 0.10.2) — the same env that loads the LeRobot data (lerobot/polars/pyarrow). No separate critic env: the OGBench-only deps (ogbench, mujoco, jax 0.6, numpy 2) are not needed offline. On a CPU/login node prefix JAX_PLATFORMS=cpu.

8. Configuration & run management

Modelled on openpi's training/config.py: a frozen, fully-typed VLAAQCConfig with nested groups, a named CONFIGS registry of documented presets, and a self-documenting run directory. Defined in vla_config.py; driven by vla_train.py (tyro CLI).

Run identity & storage

  • Self-documenting name. cfg.run_name is auto-built from the load-bearing settings, e.g. a51_sup-fixed_emb384x4L_N32_P4_b256_g0.995_s0 — atoms, support, capacity, N, prefix-grid, batch, discount, seed. So a run is identifiable from its name alone.
  • Layout (on lustre, not home). checkpoint_base_dir = /lustre/gwanwoo13/rss_post_training/phase1/critic_learning; run_dir = base/<name>/<exp_name> holds config.json (the full resolved config), metrics.csv, and checkpoints/step_XXXXXXXX/.
  • Checkpoints save params + target + optimizer state (flax msgpack) + step; pruned by keep_period; --resume restarts from the latest.
  • Logging = CSV + optional W&B (project AQC-VLA, group = config name).
