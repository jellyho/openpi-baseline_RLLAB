Run Management

VLA AQC critic 학습의 config registry / 데이터 / checkpoint / 로그 관리 규칙. 중복 job 방지 및 실험 추적 기준.

0. 데이터 상태 annotated (in-place)

원본 데이터셋이 in-place로 relabel 완료됨. data_annoation/reward_annotate.py --inplace로 디스크의 reward/mc_return 컬럼만 갱신 (rl_token/base_action은 그대로). 197GB 복사 없음.
항목값
경로 (data_root)/lustre/gwanwoo13/rss_post_training/Challenge-phase1-dataset/insert-mouse-battery_annotated
living reward−0.0004 (raw −0.0001 ×4)
failure terminal−0.5 (유지)
success terminal0.0
mc_returnγ=0.999, full-episode, 범위 [−0.5, 0]
⚠ loader 설정 주의 (이중 relabel 방지). 디스크가 이미 −0.0004라 RewardConfig.relabel_living=None, TDConfig.mc_gamma=None으로 둠. relabel을 켜면 −0.0004를 또 ×4 (→ −0.0016) 하므로 절대 켜지 말 것 (raw 데이터셋에 새로 쓸 때만). mc_gamma=None이면 디스크의 precomputed mc_return(full-episode)을 그대로 사용 (per-row-group 재계산의 truncation 회피).

1. Config Registry — vla_config.py

CONFIGS dict에 이름으로 등록된 VLAAQCConfig가 모든 run의 source of truth. 실험은 반드시 registry에 등록 후 이름으로 실행:

python vla_train.py --config <registry_key> --exp_name <optional_tag>

현재 등록된 presets

Aggregation (bootstrap V(s') 집계). TDConfig.agg_mode ∈ {max, softmax, mellowmax}, agg_beta(역온도). default max(검증된 BoN). softmax/mellowmax는 conservative 대안 — 우리 \([-1,0]\) 스케일에선 β=1이 거의 mean이라 β=4부터 시작, ablation 시 5/10/20 sweep. (상세: docs/bootstrap_aggregation.html)
⚠ vla_aqc_td_a51_stateenc run_name 충돌. state_encoder_dims가 run_name에 미포함 → default와 동일 이름. 실행 전 --exp_name stateenc 필수.

2. 디렉토리 구조 / 경로 규칙

checkpoint/로그는 scratch에 저장 (lustre 아님).

3. 실행 명령어

기본 실행

# 메인 run (TD macro, max aggregation)
python vla_train.py --config vla_aqc_td_macro

# softmax(β=4) 보수적 bootstrap ablation
python vla_train.py --config vla_aqc_td_macro_softmax

# 다른 seed / 이름 태그
python vla_train.py --config vla_aqc_td_macro --seed 1
python vla_train.py --config vla_aqc_td_macro --exp_name v1_inference_only

Resume / Throughput probe

# 마지막 checkpoint에서 이어서
python vla_train.py --config vla_aqc_td_macro --resume

# GPU throughput 측정 (checkpoint 없음) → 500k step 소요시간 추정
python vla_train.py --config vla_aqc_td_macro --timing_steps 100

4. Progress bar & 로깅 tqdm (tty-only)

표준 관행: progress bar는 터미널에서만, 로그 파일엔 bar를 안 흘림. tqdm(..., disable=not sys.stderr.isatty()) — non-tty(SLURM)에선 bar가 아예 비활성이라 carriage-return 갱신이 .out에 쌓이지 않음.
환경출력
터미널 (interactive)tqdm bar: percent / it·s⁻¹ / ETA + postfix(loss, q)
SLURM .out (non-tty)bar 없음. log_interval마다 heartbeat 한 줄 + eval 줄만
# 터미널:
vla_aqc_td_macro:  37%|███▋   | 184000/500000 [1:02:11<1:46:50, 49.3it/s, loss=0.041, q=-0.36]

# SLURM .out (heartbeat + eval, bar 스팸 없음):
  step  180000/500000 loss=0.0412 q=-0.3600 49.3 it/s
  [eval 175000] logged eval/value_curves to W&B
.out 줄 수 조절. heartbeat 빈도는 cfg.log_interval(기본 500). 더 적게 원하면 키우면 됨 (예: 5000 → 100s당 1줄). 진짜 metric 곡선은 metrics.csv/W&B가 기록하므로 .out은 liveness 확인용으로 최소만.

CSV (metrics.csv) + W&B

모든 step별 지표는 <run_dir>/metrics.csv에도 기록 (progress bar와 별개). W&B: project AQC-VLA, entity gwanwoo-yonsei-university, group=registry key, run=run_name. 네트워크 없으면 자동 skip, CSV만 남음.

Offline eval — trajectory value curves W&B eval/

롤아웃이 없으므로 eval = "기록된 trajectory 위에서 critic을 query해 mc_return과 비교"하는 시각화. 고정된 성공/실패 episode 몇 개(eval_n_success/eval_n_fail, 기본 3+3)에 대해 매 timestep Q(s_t, 실행 chunk)를 구해 mc_return(s_t)와 같은 축에 겹쳐 그림 → eval_interval마다 W&B eval/value_curves에 이미지로 logging (로컬 저장 안 함).
  • 같은 데이터셋(같은 data_root) 사용, 별도 보관 없음. episode 선택은 parquet 통계로 ~0.2초, 선택된 episode만 RAM에 한 번 캐싱(~수백 MB) 후 매 eval 재사용 → 일관된 비교.
  • 성공 traj: critic Q가 0/−0.4 쪽으로, 실패 traj: −0.5 쪽으로 떨어지면 calibration 양호.
  • 코드: vla_eval.py. standalone: python vla_eval.py --config <k> --step <n> --out fig.png (또는 --wandb).

주요 모니터링 지표 (train)

지표기대설명
train/critic_loss↓HL-Gauss CE; 수렴 시 ~0.05 이하
train/q_mean−0.3 ~ −0.4예측 Q 평균 (설계 시작 value)
train/target_mean−0.3 ~ −0.4TD target 평균
train/dist_target_oob= 0.0target이 \([-1,0]\) 밖이면 >0 → 즉시 점검
train/dist_edge_mass↓support 경계 atom에 분포 쌓이면 support 재검토
train/term_frac~0.001terminal 비율; 비정상으로 높으면 false-terminal 의심
train/v_next_mean−0.3 ~ −0.4bootstrap V(s') 평균 (agg_mode 영향 — softmax면 max보다 낮음)

정성 평가는 위 eval/value_curves(W&B 이미지)로 — 스칼라 q_vs_mc 상관은 제거하고 trajectory 곡선으로 대체.

6. 중복 job 방지 체크리스트

  1. 실행 전: run_dir/config.json 존재 = 같은 config run 있음. checkpoints/ 확인 후 --resume 결정.
  2. hparam 변경: run_name 자동 변경 → 새 폴더 → 충돌 없음.
  3. run_name에 반영되는 변경: atoms / support / arch / N / prefixes / batch / discount / agg_mode·agg_beta(soft일 때) / reward_normalize / commander_filter / seed.
  4. run_name에 반영 안 되는 변경 (→ --exp_name 필수):
    • state_encoder_dims — --exp_name stateenc
    • terminal_uses_mc
    • relabel_living/mc_gamma (현재 둘 다 None 고정이라 사실상 무관)
  5. SLURM 동시 실행: 같은 run_dir 두 job이 쓰면 checkpoint 충돌 → 다른 --exp_name 사용.
